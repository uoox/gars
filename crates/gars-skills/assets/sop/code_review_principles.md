# Code Review Principles

## 4 问自检

每次准备提交代码前问自己：

1. 这段改动的**职责**是否单一？（增删的文件/函数都属于同一个目标）
2. **局部可推理**吗？读这块代码不用跳到 N 个远处文件就能理解？
3. **失败半径**多大？出错时坏掉多少东西？是否可以缩小到一个模块？
4. **可验证**吗？我能用具体的测试或一次工具调用证明它行？

## 14 条原则

### 模块层面

1. **职责单一**：一个文件只做一件事；多事项必拆分。
2. **依赖向核心**：边缘模块依赖核心，反之是坏味道。
3. **接口收敛**：暴露最少的 public 名字，能 pub(crate) 就别 pub。
4. **可组合**：函数返回值能塞进下一个函数，少用 side-effect。

### 函数层面

5. **短小**：函数 >80 行先想一想能不能拆。
6. **少参数**：>5 个参数考虑结构体；不要塞进无关上下文。
7. **错误显式**：用 `Result`、`thiserror`，不藏 panic、不吞错误。
8. **早返回**：错误条件用 `return Err` / `?` 早走，主路径不缩进。

### 修改层面

9. **最小变更**：一次只解决一个问题，不顺手大改。
10. **测试同步**：能加测试的修改必须加。
11. **可逆**：不可逆操作要有备份/dry-run。
12. **不引战**：风格保持与上下文一致，不强行 lint 改写。

### 评审层面

13. **读者优先**：注释解释 WHY，不解释 WHAT。
14. **审查证据**：提交时附上跑过的命令/测试输出/截图，让评审人少猜。
